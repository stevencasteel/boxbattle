1. RUNTIME STACK AND CONFIGURATION BOUNDARY

- React 19
- TypeScript 6
- Zustand 5
- Vite 8
- HTML5 Canvas for gameplay rendering
- Framer Motion for UI transitions
- canvas-confetti for end-of-match overlays
- lucide-react for iconography

2. BOOT, FIRST-GESTURE, AND APP INITIALIZATION
───────────────────────────────────────────────────────────────────────────────

Relevant files:
- `src/main.tsx`
- `src/App.tsx`
- `src/hooks/useBootSequence.ts`
- `src/hooks/useFirstGesture.ts`
- `src/components/GameArena.tsx`

Startup begins in `main.tsx`, which mounts the React root in `StrictMode`.
The app then moves through its own boot logic before the gameplay engine is
allowed to run.

Core initialization responsibilities:
- mount the application shell
- restore save and settings state
- attach the current input map
- wait for the first user gesture before starting audio playback
- construct the engine only when the canvas is available

The browser audio gate is important. Audio cannot reliably start until a user
gesture occurs, so the app uses a first-interaction hook to unlock playback and
then initialize the procedural audio graph.

Typical startup sequence:

[React root mounts]
        |
        v
[App shell and menu state render]
        |
        v
[Boot / save / settings initialization]
        |
        v
[first pointer or keyboard gesture]
        |
        v
[audio graph initializes]
        |
        v
[engine starts and gameplay canvas becomes active]

This architecture keeps menus responsive while avoiding autoplay restrictions.

3. WORLD, ENTITY, AND COMPONENT ARCHITECTURE
─────────────────────────────────────────────────────────────────────────

Relevant files:
- `src/core/World.ts`
- `src/core/Interfaces.ts`
- `src/entities/BaseEntity.ts`
- `src/entities/components/*`
- `src/entities/Player.ts`
- `src/entities/Boss.ts`
- `src/entities/Projectile.ts`
- `src/entities/BaseMinion.ts`

The engine uses a component-oriented entity architecture:
- `World` owns the simulation surfaces and services
- entities encapsulate identity, transform, and behavior
- components hold discrete capabilities such as health, physics, melee, dash,
  fireball, and input reception

World-level responsibilities:
- store the player, boss, and active minions
- provide the physics world
- provide the event bus
- provide the audio interface
- provide the input interface
- manage the projectile pool

Entity state is kept mostly off React. The simulation layer updates entities
directly, while a projection system copies only the necessary high-level state
into the UI store.

Primary entity categories:
- player: movement, combat, heal charge, determination
- boss: phase-based AI, projectile logic, cinematic transitions
- projectile: pooled owner-specific shot with damage and lifespan
- minions: lightweight enemy actors with distinct movement and attack patterns
- spawners: fixed points that instantiate minions during a battle

Components are used to separate behavior from identity. The codebase includes
component types such as:
- `PhysicsComponent`
- `HealthComponent`
- `MeleeComponent`
- `DashComponent`
- `FireballComponent`
- `HealComponent`
- `InputReceiverComponent`

This keeps the architecture modular without requiring a full ECS framework.

4. MAIN LOOP, FIXED-STEP SIMULATION, AND PROJECTION
───────────────────────────────────────────────────────────────────────────────

Relevant files:
- `src/core/GameLoop.ts`
- `src/core/Engine.ts`
- `src/core/StateProjectionSystem.ts`
- `src/core/SimulationSystems.ts`

The runtime loop uses requestAnimationFrame for display cadence, but simulation
is normalized around a fixed update step.

Observed loop behavior:
- frame delta is measured from the previous RAF tick
- delta is clamped to prevent runaway stalls
- the engine advances simulation using a 60 Hz canonical step
- the render pass follows the update pass
- visibility changes stop and restart the loop

The `GameLoop` class clamps very large frame deltas to 0.1 seconds. The `Engine`
maintains a fixed-step accumulator internally and runs at 1/60 second
simulation slices.

A useful mental model:

[RAF tick]
   -> measure dt
   -> clamp dt
   -> advance world state
   -> render current frame

State projection is intentionally narrow. The `StateProjectionSystem` only
pushes changes that matter to the UI:
- player HP
- boss HP
- healing charges
- determination counter

It also tracks a crisis timer. When the player's HP drops to 1 from a higher
value, the system triggers a crisis window of 0.45 seconds. That timer can be
used by presentation code to intensify effects or slow the game flow.

The result is a split between:
- authoritative simulation state in the engine
- derived presentation state in Zustand

───────────────────────────────────────────────────────────────────────────────
5. PHYSICS WORLD AND COLLISION BROAD PHASE
───────────────────────────────────────────────────────────────────────────────

Relevant files:
- `src/core/PhysicsWorld.ts`
- `src/core/Interfaces.ts`
- `src/core/Units.ts`

Static geometry is partitioned into three indexed collections:
- solids
- one-way platforms
- hazards

Each group is stored in its own spatial hash grid. The cell size is 200 pixels,
matching `UNITS.SPATIAL_GRID_CELL_SIZE`.

Bucket keys are generated with:
- `key = (cy << 16) | cx`

That layout supports fast broad-phase lookup without full pairwise scanning.

The broad-phase pipeline is:
1. compute the query AABB
2. map the AABB into grid cells
3. collect candidate rectangles from the matching buckets
4. perform narrow-phase overlap checks only on the candidates

This is applied separately for:
- solid walls and floors
- one-way platforms
- hazards

The engine also uses conservative movement handling for fast actors:
- swept collision stepping for dashes and projectiles
- a small corner-nudge allowance so the player does not snag on lip edges
- platform logic that only resolves one-way surfaces when the actor is moving
  downward and approaching from above

The architecture is designed to keep collision cost low while preserving the
feel of a fast arcade fighter.

6. INPUT MAPPING AND CONTROL SURFACE
───────────────────────────────────────────────────────────────────────────────

Relevant files:
- `src/core/InputProvider.ts`
- `src/core/SettingsManager.ts`
- `src/core/menuNavigation.ts`
- `src/store/useGameStore.ts`

The input layer supports:
- keyboard
- gamepad
- configurable key maps
- a pause edge trigger on `KeyP`

Canonical actions:
- MOVE_LEFT
- MOVE_RIGHT
- MOVE_UP
- MOVE_DOWN
- JUMP
- ATTACK
- DASH

Default presets:
- Default 1: arrow keys plus A/D/W/S, Space/X, C, Z
- Default 2: A/D/W/S, Period, Comma, Slash
- Custom: user-rebound map saved in settings

Menu navigation reuses the same action map. Confirm and back behavior are
resolved through helper logic rather than hard-coded key names.

Persistence details:
- the current settings blob is stored under `box_battle_config`
- validation is schema-based and tolerant of malformed or partial data
- a failed validation falls back to safe defaults

This keeps the game playable even if localStorage is corrupted or stale.

7. PLAYER TRAVERSAL AND COMBAT MECHANICS
───────────────────────────────────────────────────────────────────────────────

Relevant files:
- `src/entities/Player.ts`
- `src/entities/handlers/PlayerInputHandler.ts`
- `src/entities/handlers/PlayerCombatHandler.ts`
- `src/entities/components/DashComponent.ts`
- `src/entities/components/MeleeComponent.ts`
- `src/entities/components/FireballComponent.ts`
- `src/entities/components/HealComponent.ts`

The player controller is tuned for responsive arcade movement rather than
physically literal motion.

Traversal features:
- horizontal acceleration and deceleration
- jump handling
- wall-slide behavior
- wall jump rebound
- dash with duration and cooldown
- coyote timing and jump buffering
- optional double-jump behavior if enabled by the current ability state

The common movement pattern is:
- read the input state
- compute target horizontal speed
- apply acceleration or friction
- resolve physics and collision constraints
- project the result into renderable state

Combat features:
- horizontal melee sweep
- upward melee sweep
- downward pogo strike
- projectile firing with charge tiers
- healing charge / determination mechanics

Melee resolution is range-sensitive:
- the close-range sweet spot rewards precision with higher damage
- the normal sweep deals baseline damage
- the downward strike has its own hitbox geometry and rebound behavior

Determination and healing:
- successful hits contribute to a counter used for healing
- the heal action is a timed channel
- the channel consumes a charge and restores health on completion
- interruption occurs if the player takes damage during the channel

The design strongly favors readable player feedback:
- hit sparks
- screen shake
- brief hit-stop windows
- deterministic collision responses

───────────────────────────────────────────────────────────────────────────────
8. PROJECTILE OWNERSHIP AND COMBAT RESOLUTION
───────────────────────────────────────────────────────────────────────────────

Relevant files:
- `src/entities/Projectile.ts`
- `src/entities/ProjectileStrategy.ts`
- `src/core/ObjectPool.ts`
- `src/core/World.ts`

Projectiles are pooled rather than instantiated and destroyed repeatedly.
That reduces garbage generation and keeps hot-path performance stable.

Ownership model:
- player-owned projectiles
- boss-owned projectiles

The owner determines:
- valid collision targets
- visual treatment
- whether projectile clash behavior is allowed
- base damage and movement behavior

Projectile lifecycle:
- spawn from the world’s projectile pool
- travel for a finite lifespan
- collide with valid targets or surfaces
- return to the pool when spent

The codebase treats projectiles as a generic combat primitive, but the
strategy layer keeps boss shots and player shots distinct in behavior and
presentation.

9. BOSS AI, PHASE GATES, AND CINEMATIC ESCALATION
───────────────────────────────────────────────────────────────────────────────

Relevant files:
- `src/entities/Boss.ts`
- `src/entities/BossStates.ts`
- `src/core/BattleDirector.ts`
- `src/core/CinematicSystem.ts`

The boss is phase-driven. The battle director watches the boss HP and triggers
narrative or mechanical shifts when threshold conditions are crossed.

Phase gates:
- Phase 1: above 70% HP
- Phase 2: 70% to 40% HP
- Phase 3: below 40% HP

Observed escalation behavior:
- first boss damage can trigger introductory dialogue
- phase 2 publishes a phase-shift event and unlocks more aggressive attacks
- phase 3 publishes another phase-shift event and pushes final-stand behavior

The boss’s movement and attack selection are state-machine driven:
- patrol or cooldown states
- telegraph state
- lunge or attack state
- post-impact recovery state

The battle director is also responsible for kicking off cinematic handling when
the boss dies or the match ends. That keeps victory/defeat presentation out of
the low-level combat code.

Relevant constants from the configuration layer include:
- BOSS_PATROL_SPEED_BASE = 160
- BOSS_LUNGE_SPEED_BASE = 960
- BOSS_WALL_REBOUND_VELOCITY = 280
- BOSS_LUNGE_DAMAGE = 2

The important architectural point is that AI decides intent, while physics and
collision decide outcome.

10. MINIONS, SPAWNERS, AND SPAWN-CAGE PRESENTATION
───────────────────────────────────────────────────────────────────────────────

Relevant files:
- `src/entities/BaseMinion.ts`
- `src/entities/TurretMinion.ts`
- `src/entities/LancerMinion.ts`
- `src/entities/FlyerMinion.ts`
- `src/entities/Spawner.ts`
- `src/core/visuals/MinionVisuals.ts`

The minion system is composed of a few distinct archetypes:
- TurretMinion: stationary ranged pressure
- LancerMinion: patrol and lunge behavior
- FlyerMinion: oscillating vertical or aerial pressure

Spawners place minions at fixed encounter points defined by level data.

Spawn presentation:
- newly spawned minions can be held in a brief non-collidable window
- a stylized cage effect is rendered during that window
- the cage uses segmented wireframe geometry and squash-style deformation

This effect is cosmetic and communicates spawn timing to the player. It is not
a separate gameplay entity.

The minion architecture is intentionally lighter than the boss architecture. The
goal is readable encounter pressure rather than fully independent AI systems for
every enemy type.

11. CAMERA, SHAKE, AND HIT-STOP
───────────────────────────────────────────────────────────────────────────────

Relevant files:
- `src/core/Camera.ts`
- `src/core/CinematicSystem.ts`

The game does not use camera panning or camera zoom as part of its core
presentation. The screen stays fixed and readable, with motion delivered through
impact feedback instead.

Camera-related behavior includes:
- screen shake / tremor
- directional shake for strong impacts
- decay over time
- event-driven intensity changes

High-impact events can also trigger hit-stop windows. Those pauses help sell
combat weight and make high-value attacks feel more forceful.

The architecture separates:
- steady view framing
- transient shake effects
- cinematic freeze or slow-down effects

That separation prevents the presentation layer from turning into a hidden
source of simulation logic.

12. RENDERING PIPELINE AND VISUAL EFFECTS
───────────────────────────────────────────────────────────────────────────────

Relevant files:
- `src/core/WorldRenderer.ts`
- `src/core/EntityRenderer.ts`
- `src/core/ParticleRenderer.ts`
- `src/core/StaticMapRenderer.ts`
- `src/core/effects/CinematicDeathRenderer.ts`
- `src/core/effects/PlayerFxRenderer.ts`

Rendering is canvas-first and procedurally driven.

Primary render responsibilities:
- draw static map geometry
- draw entities
- draw particles
- draw cinematic overlays
- draw UI-adjacent world effects

Renderer structure:
- `WorldRenderer` orchestrates the frame
- `StaticMapRenderer` caches and draws static geometry
- `EntityRenderer` draws actors and their visual transforms
- `ParticleRenderer` handles sparks, rings, dust, and blast events

Visual styles:
- squash and stretch on key impacts
- glow and gradient effects
- procedural rings and sparks
- transient trails and impact flashes

The renderer favors a low-allocation strategy. Reusable objects and pools are
used where practical so the hot loop stays stable under action-heavy scenes.

13. PROCEDURAL AUDIO AND AUDIO LIFECYCLE
───────────────────────────────────────────────────────────────────────────────

Relevant files:
- `src/core/SoundSynth.ts`
- `src/core/audio/AudioContextManager.ts`
- `src/core/audio/MusicSequencer.ts`
- `src/core/audio/SFXManager.ts`
- `src/core/audio/DroneManager.ts`
- `src/core/audio/sfx/*`
- `docs/MUSIC_ARCHITECTURE.txt`

Audio is procedural rather than asset-driven:
- oscillators
- filters
- envelope shaping
- gain staging
- transport or lifecycle control

The audio pipeline is gesture-gated by the browser. Once unlocked, the app
initializes the audio graph and then plays the appropriate menu or gameplay
music.

Observed audio architecture:
- separate gain paths for music and SFX
- cabinet-style filtering on the output chain
- limiter protection on the final output
- dynamic pitch scaling for combo feedback
- procedural drones for healing and charge states

The music architecture document describes C Dorian pitch ratios and combo-linked
pitch stepping. That is consistent with the current procedural design: hit
confirms can be tied to a repeatable, musically coherent scale chain.

Important practical rule:
- do not depend on heavy binary audio assets for core gameplay feedback
- generate the important sounds in code

14. SAVE DATA, SETTINGS, AND VALIDATION
───────────────────────────────────────────────────────────────────────────────

Relevant files:
- `src/core/SaveManager.ts`
- `src/core/SettingsManager.ts`
- `src/core/schemas.ts`
- `src/store/useGameStore.ts`

Save slots:
- three slots are stored in localStorage
- slot data includes wins, losses, and empty state
- copying and erasing slots are supported operations

Save storage key:
- `box_battle_save_slots`

Settings storage key:
- `box_battle_config`

Validation is explicit and defensive:
- save slot values are sanitized to non-negative integers
- audio settings are clamped or replaced with fallbacks
- key maps are validated against the required action list

The UI store separates:
- session state
- gameplay state
- screen state
- result state

That separation keeps the menu system from mutating core simulation logic
directly.

The main screens visible in the store-level state are:
- TITLE
- SAVE_SELECT
- OPTIONS
- SOUND
- CONTROLS
- CREDITS
- SOURCE_VIEW
- PLAYING

The title flow and result flow are handled through store state and screen route
helpers, not through hard-wired page changes.

15. BUILD, MANIFEST GENERATION, AND DEPLOYMENT

Pre-dev and pre-build hooks:
- generate the source manifest
- generate the consolidated source context file

The build pipeline is:
1. lint
2. TypeScript compile
3. Vite production bundle

Deployment automation:
- push to `main`
- build in GitHub Actions
- deploy to GitHub Pages
- push the same build artifact to Itch.io via Butler

The manifest generation workflow is also part of the project’s documentation
story. It allows the source code to be flattened into a single searchable context
document for AI-assisted debugging and review.
